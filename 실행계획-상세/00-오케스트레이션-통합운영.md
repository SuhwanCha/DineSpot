## 1. 공통 기술 스택 (고정)

## Frontend
- Next.js (App Router), TypeScript, Tailwind CSS
- 상태관리: TanStack Query + Zustand
- 컴포넌트 규칙: Atomic 기반(Atoms/Molecules/Organisms)
- 폼/검증: React Hook Form + Zod
- 테스트: Vitest, React Testing Library, Playwright(E2E)

## Backend
- Go 1.23+, gRPC, grpc-gateway(외부 공개 API 필요 시)
- 인증/인가: JWT + gRPC interceptor + RBAC 매트릭스
- 비동기 이벤트: Redis Streams (기본), 필요 시 Kafka 확장 여지
- 테스트: Go test(unit/integration), k6(perf)

## Data/Infra/Observability
- MySQL 8.x (주 저장소), Redis 7.x (캐시/큐/락)
- Docker Compose(로컬), GitHub Actions(CI), Terraform(IaC)
- 모니터링: Prometheus + Grafana + Loki + OpenTelemetry Trace

## 2. 아키텍처 오케스트레이션 원칙

1. **계약 우선**: 기능 코딩 전에 proto/OpenAPI/이벤트 스키마를 먼저 확정.
2. **주간 통합**: 최소 주 2회(수/금) main 브랜치 통합.
3. **호환성 유지**: DB 마이그레이션은 backward-compatible 2단계 배포(추가→전환→정리).
4. **기능 플래그**: 미완성 기능은 flag로 숨기고 main 통합 허용.
5. **관측성 기본 탑재**: 신규 API는 로그/메트릭/트레이스 3종 의무.

---

## 3. 브랜치/릴리스/계약 관리

- 브랜치 전략: `main`(배포가능) / `develop`(통합) / `feature/<owner>-<topic>`
- PR 규칙:
  - 작은 단위(권장 300라인 이하)
  - 최소 1명 타 도메인 리뷰 필수(교차 검증)
  - 체크 통과: lint, unit test, contract test
- 계약 버전:
  - gRPC/proto: SemVer 태그 (`v1alpha`, `v1`)
  - 이벤트 스키마: schema registry 문서화 (필드 추가만 허용)

## 4. 주간 통합 이벤트 캘린더 (고정)

- **월요일 오전 — 인터페이스 동기화(60분)**
  - A~E: 이번 주 API/이벤트/스키마 변경 예정 발표
  - 결과물: `Interface Change Log` 갱신
- **수요일 오후 — 중간 통합 리허설(90분)**

  - 실패 시 담당자 즉시 핫픽스 브랜치 생성
- **금요일 오후 — 릴리스 후보 점검(120분)**
  - E2E 핵심 시나리오 + SLO 점검
  - 다음 주 리스크/차단 이슈 확정

## 6. 충돌 방지용 인터페이스 책임 매트릭스

- **A 제공**: 사용자/권한 컨텍스트, 토큰 검증 미들웨어
- **B 제공**: 매장/메뉴/좌석/영업시간 조회 계약
- **C 제공**: 예약 상태 머신, 가용 슬롯 계산 계약
- **D 제공**: 웨이팅 순번/호출 이벤트 계약
- **E 제공**: 감사로그, KPI 집계, 알람 규칙

교차 의존성 승인권:
- A↔C: 인증 주체/권한 필드 변경은 양측 승인 필수
- B↔C: 좌석 정책/운영시간 포맷 변경은 양측 승인 필수
- C↔D: 예약상태 이벤트명 변경은 양측 승인 필수
- 전원↔E: 로그 필드/메트릭 명칭 변경은 E 승인 필수

## 7. 통합 실패 시 즉시 대응 플로우

1. 통합 실패 탐지(CI 또는 수동 리허설)
2. 원인 도메인 owner 지정(30분 이내)
3. hotfix 브랜치 생성 및 2시간 내 재시도
4. 실패가 2회 반복되면 범위 축소/flag 처리
5. 금요일 회고에 재발방지 액션 등록

---

## 8. 공통 완료 기준 (통합 관점)

- 각 도메인 PR의 contract test 전부 green
- 핵심 E2E 5개 시나리오 모두 통과
- P1/P2 버그 잔여 0
- SLO: 예약 API p99 200ms 이내, 웨이팅 이벤트 전파 1초 이내
- 운영 문서(runbook/rollback/on-call) 최신화
