# 레스토랑 예약 서비스 10주 개발 실행 계획 (담당자 A~E)

> 기준일: 2026-03-24  
> 기간: 총 10주 (Week 1 ~ Week 10)  
> 인력 구성: PO/PM(작성자) + 개발 담당 5명(A, B, C, D, E)  
> 운영 원칙: 각 담당자는 본인 Feature를 **개발 → QA → 배포 준비**까지 End-to-End로 책임

---

## 1) 목표 및 운영 원칙

## 프로젝트 목표
- 10주 내 MVP 출시 가능한 수준의 레스토랑 예약 플랫폼 완성
- 고객/매장/관리자 핵심 시나리오를 단절 없이 제공
- 기능 단위 오너십 기반으로 개발 속도와 품질 동시 확보

## 팀 운영 원칙
- **Feature Owner 제도**: A~E가 각자 기능 전 생명주기 책임
- **Definition of Done(DoD)**: API 명세, 코드, 테스트, 운영 가이드까지 완료해야 종료
- **주간 게이트 리뷰**: 매주 금요일, 개발/QA 상태를 수치로 점검
- **품질 기준**: 단위/통합/E2E 테스트, 장애 대응 시나리오, 모니터링 지표 포함

## 2) 담당 Feature 매핑 (A~E)

| 담당자 | Feature Domain | 책임 범위 |
|---|---|---|
| A | 사용자 인증/권한/프로필 | 회원가입, 로그인, JWT, 권한 모델, 사용자 프로필, 접근 제어 |
| B | 레스토랑/메뉴 관리 | 매장 등록/수정, 메뉴 CRUD, 영업시간/좌석 정책, 멀티테넌시 데이터 정책 |
| C | 예약(Reservation) 코어 | 예약 생성/변경/취소, 슬롯 가용성, 중복 예약 방지, 예약 상태 머신 |
| D | 웨이팅/알림 | 실시간 웨이팅 큐, 입장 처리, 순번 변동, 알림(SMS/Email/푸시 연동 포인트) |
| E | 관리자/운영/관측성 | 관리자 승인 플로우, 대시보드, 로그/메트릭/알람, 운영 도구 |

> 공통(전원 협업): 프론트엔드 디자인 시스템, 공통 컴포넌트, CI/CD, 보안 점검, 릴리스 리허설

## 3) 10주 마스터 일정 (주차별)

| 주차 | 목표 | 산출물 |
|---|---|---|
| Week 1 | 킥오프/요구사항 확정/개발 착수 | 우선순위 백로그, 공통 개발 규칙, 스프린트 목표 확정 |
| Week 2 | 개발 환경/공통 모듈/골격 구현 | 모노레포 구조, 인증/로깅/에러핸들링 공통 모듈, CI 파이프라인 1차 |
| Week 3 | 핵심 기능 개발 1차 (A/B/C 중심) | 인증/매장/예약 주요 API, 기본 UI 플로우 연결 |
| Week 4 | 핵심 기능 개발 2차 (D/E 중심) | 웨이팅/알림, 관리자 승인/대시보드 1차 |
| Week 5 | 기능 통합 및 시나리오 연결 | 고객→매장→관리자 End-to-End 연계, 통합 테스트 시작 |
| Week 6 | 기능 완성 및 QA 준비 | 잔여 기능 마감, 테스트 데이터/시나리오 고도화 |
| Week 7 | 안정화 1차 (성능/보안/예외) | 부하 테스트 결과, 취약점 점검 결과, 장애 대응 플로우 |
| Week 8 | QA 집중 주간/UAT 준비 | 결함 수정, 회귀 테스트, UAT 시나리오/데이터 준비 |
| Week 9 | UAT/출시 후보(RC) 확정 | RC 빌드, 운영 체크리스트, 롤백 계획 |
| Week 10 | 출시/초기 운영 안정화 | 운영 전환, 핫픽스 체계, 1주차 운영 리포트 |

## 4) 담당자별 상세 실행 계획 (개발-QA)

## A — 사용자 인증/권한/프로필
- **개발 (W1~W6)**
  - Auth/User API 및 미들웨어 구현
  - 권한 기반 라우트 가드(고객/사장/관리자) 적용
  - 사용자 프로필 및 계정 관리 화면/API 연동
- **QA (W7~W9)**
  - 인증 예외 케이스(만료/위조/권한 없음) 테스트
  - 보안 테스트(OWASP Top10 관련 항목 점검)
  - 로그인/권한 회귀 테스트 자동화

## B — 레스토랑/메뉴 관리
- **개발 (W1~W6)**
  - 매장 CRUD, 메뉴 CRUD, 운영시간/좌석 설정 API 구현
  - 사장용 관리 UI 및 입력 검증 로직 구현
  - 이미지 업로드 및 리소스 참조 구조 연동
- **QA (W7~W9)**
  - 테넌트 간 데이터 격리 테스트
  - 대량 메뉴/매장 데이터에서 성능 점검
  - 매장 등록→수정→비활성화 전 과정 회귀 테스트

## C — 예약 코어
- **개발 (W1~W6)**
  - 예약 생성/조회/변경/취소 API 구현
  - 좌석 가용성 계산 및 충돌 방지 로직 구현
  - 고객 예약 화면 및 사장 승인 화면 연결
- **QA (W7~W9)**
  - 동시 예약 경쟁 시나리오 부하 테스트
  - 실패 복구(부분 실패/재시도) 검증
  - 예약 핵심 시나리오 E2E 자동화

## D — 웨이팅/알림
- **개발 (W1~W6)**
  - 웨이팅 큐 API 및 순번 계산 로직 구현
  - 실시간 순번 업데이트 및 점주 호출 기능 구현
  - 외부 알림 채널 연동 인터페이스 구현
- **QA (W7~W9)**
  - 순번 정합성/지연 시간 측정
  - 네트워크 불안정 상황 재연 테스트
  - 알림 실패/재전송/중복 방지 검증

## E — 관리자/운영/관측성
- **개발 (W1~W6)**
  - 관리자 승인/제재/문의 대응 기능 구현
  - 운영 대시보드(차트/필터/기간 비교) 개발
  - 로그/메트릭/트레이싱 수집 파이프라인 구축
- **QA (W7~W9)**
  - 권한 분리 및 감사 로그 무결성 검증
  - 알람 민감도 튜닝(오탐/미탐 최소화)
  - 장애 모의훈련(GameDay) 및 운영 Runbook 검증

## 6) 완료 기준 (Definition of Done)

각 Feature(A~E)는 아래 조건을 모두 만족해야 완료로 인정:
1. 코드 구현 완료 및 코드리뷰 승인 2인 이상
2. 테스트 통과(단위/통합/E2E) + 주요 성능 지표 충족
3. 운영 문서(Runbook, 장애 대응, 대시보드 지표) 작성
4. 릴리즈 노트 및 알려진 이슈 등록
